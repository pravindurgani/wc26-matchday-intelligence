## CRIT #1 — `refreshLive()` circuit-breaker scan was string-only; Python emits objects

### What was wrong (v2.3.3)

`run_live_update.py` writes warnings as **objects** at three sites:
- `run_live_update.py:533` — `{"type": "circuit_breaker", "message": "consecutive failures..."}`
- `run_live_update.py:556` — same shape on retry-exhaustion
- `run_live_update.py:570` — same shape on rate-limit-hit

`refreshLive()` in v2.3.3 scanned with:

```js
const cbTripped = warnArr.some(function(w) {
  return String(w || '').toLowerCase().indexOf('circuit_breaker') !== -1;
});
```

`String({type:'circuit_breaker',...})` returns the literal string `"[object Object]"`, which does NOT contain `circuit_breaker`. Every CB warning therefore evaded detection, and `cbTripped` was permanently `false`. The conservative branch that holds `staleKill` open while the upstream pipeline is degraded never fired — the engine would silently re-enable live updates the moment age dropped below the threshold, even with the Python side still in CB cooldown.

### What v2.3.4 does

```js
const cbTripped = warnArr.some(function(w) {
  if (typeof w === 'string') {
    return w.toLowerCase().indexOf('circuit_breaker') !== -1;
  }
  if (w && typeof w === 'object') {
    const t = String(w.type || '').toLowerCase();
    const m = String(w.message || '').toLowerCase();
    return t.indexOf('circuit_breaker') !== -1 ||
           m.indexOf('circuit_breaker') !== -1 ||
           m.indexOf('circuit breaker') !== -1;
  }
  return false;
});
```

Three matchers (type field, message snake-case, message space-form) ensure CB is detected whether the producer evolves to drop the type field, switches to human-readable phrasing, or reverts to a bare string.

### Validation receipts

`test_harness.mjs › v234_regressions.fixtures`:
- `crit1_object_type_field: ok` — `{type:'circuit_breaker'}` detected
- `crit1_object_message_only: ok` — `{message:'...circuit_breaker...'}` detected
- `crit1_object_circuit_breaker_with_space: ok` — `{message:'circuit breaker open'}` detected
- `crit1_string_legacy: ok` — bare string still works (back-compat)
- `crit1_no_warnings: ok` — empty list returns false
- `crit1_unrelated_object: ok` — `{type:'foo'}` returns false

## HIGH #2 — `warnings.join()` rendered objects as `"[object Object]"`

### What was wrong (v2.3.3)

The same warnings array that fed CRIT #1 was also rendered to `Live!intelWarn` via `warnArr.join(' | ')`. With object payloads that produced cells like:

```
[object Object] | [object Object] | [object Object]
```

— useless on a phone, useless for the operator on glance, and breaks the convention that every warning the engine surfaces is human-readable.

### What v2.3.4 does

Added `_renderWarn_()` close to the `refreshLive()` site:

```js
function _renderWarn_(w) {
  if (w == null) return '';
  if (typeof w === 'string') return w;
  if (typeof w === 'object') {
    if (w.message) return String(w.message);
    if (w.type) return String(w.type);
    try { return JSON.stringify(w); } catch (e) { return String(w); }
  }
  return String(w);
}
```

Render precedence: `message` field (rich human text) → `type` field (slug) → JSON snapshot (last-ditch debug). String/null/scalar fall through unchanged.

### Validation receipts

- `high2_object_message_render: ok` — `{message:'X'}` → `'X'`
- `high2_object_type_fallback: ok` — `{type:'foo'}` → `'foo'`
- `high2_no_object_object_literal: ok` — no path produces `[object Object]`
- `high2_mixed: ok` — `[str, {message:'A'}, {type:'B'}]` → `'str | A | B'`

## HIGH #3 — `_writeCalibratedProbs_` fallback substituted 0% for "no signal"

### What was wrong (v2.3.3)

When the calibration LUT returned an empty-string sentinel (the "no bins" / "out-of-domain" marker), `_writeCalibratedProbs_` accepted the row anyway because:

```js
const nh = Number(ch);            // Number('') === 0
isFinite(nh)                      // isFinite(0) === true
```

The fallback guard intended to detect non-finite output silently let `0` through. The downstream sheet wrote `0.000` for the calibrated prob, which the Bets sheet interpreted as "this outcome will never happen" — a hard zero that nukes any Kelly stake, instead of a graceful "fall back to raw" signal.

Root cause is in `_interp_()` itself: the `isFinite(p)` guard is satisfied by `Number('')===0`, so the early-return at `bins[0].actual_freq` fires on bottom of the curve.

### What v2.3.4 does

`_interp_()` now rejects sheet-empty markers BEFORE the numeric guard, and uses the coerced numeric throughout:

```js
function _interp_(p, bins) {
  if (p === '' || p === null || p === undefined) return '';
  const n = Number(p);
  if (!isFinite(n)) return '';
  if (n <= bins[0].mean_pred) return bins[0].actual_freq;
  if (n >= bins[bins.length - 1].mean_pred) return bins[bins.length - 1].actual_freq;
  for (let i = 0; i < bins.length - 1; i++) {
    const a = bins[i], b = bins[i + 1];
    if (n >= a.mean_pred && n <= b.mean_pred) {
      const t = (n - a.mean_pred) / (b.mean_pred - a.mean_pred);
      return a.actual_freq + t * (b.actual_freq - a.actual_freq);
    }
  }
  return n;
}
```

`_writeCalibratedProbs_` fallback now correctly detects `''` (along with non-finite) and returns the RAW row `[rh, rd, ra]`, not `[0, 0, 0]`.

### Validation receipts

- `high3_interp_empty_string_on_non_finite: ok` — `_interp_('', bins)` returns `''` (not 0.1, the bottom-of-curve value)

The test note in the harness is the load-bearing audit line: *"Number(\"\") is 0 so isFinite says yes — guard MUST also check ch !== \"\""*.

## HIGH #4 — Calibration silently clamped out-of-bin inputs

### What was wrong (v2.3.3)

If a raw prob landed outside the calibration table's domain (e.g. draw=0.34 with bins capped at 0.30), `_interp_` silently clamped to the edge — returning `bins[last].actual_freq` for every value above the range. The operator had no signal that the engine was extrapolating.

### What v2.3.4 does

New helper `_isOutOfBinRange_(p, bins)` flags out-of-domain inputs:

```js
function _isOutOfBinRange_(p, bins) {
  if (!Array.isArray(bins) || bins.length === 0) return true;
  const n = Number(p);
  if (!isFinite(n)) return true;
  return n < bins[0].mean_pred || n > bins[bins.length - 1].mean_pred;
}
```

`_writeCalibratedProbs_` now counts out-of-range hits per side (home/draw/away) and surfaces the count via the existing diagnostic cell. The clamp behaviour of `_interp_` is preserved (don't break working pipelines), but the silence is gone.

### Validation receipts

- `high4_below_range: ok` — `_isOutOfBinRange_(0.02, [0.05..0.30])` → `true`
- `high4_above_range: ok` — `_isOutOfBinRange_(0.5, [0.05..0.30])` → `true`
- `high4_in_range: ok` — `_isOutOfBinRange_(0.2, ...)` → `false`
- `high4_at_low_edge: ok` — `p == bins[0].mean_pred` → `false` (boundary inclusive)
- `high4_at_high_edge: ok` — `p == bins[last].mean_pred` → `false`
- `high4_empty_bins: ok` — empty array → `true` (no domain to be in)

## MED #5 — Strict date validation in `refreshLive()` (`generated_at`)

### What was wrong (v2.3.3)

`refreshLive()` parsed `generated_at` via `new Date(raw)` and gated `staleKill` on `isFinite(ageMin)`. But:

- `new Date('null')` → `Invalid Date` → `getTime()` is `NaN`
- The else-branch (`if (cbTripped || !isFinite(ageMin))`) was inverted in the original: `cbTripped=false` + `!isFinite(ageMin)=true` should hold `staleKill` open, but the v2.3.3 wiring fell through to the "clear staleKill" path on the literal string `'null'`.

This meant a Python-side emitter that wrote the JSON string `"null"` (rather than the JSON literal `null`) for `generated_at` would silently clear the stale-kill flag — same failure shape as CRIT #1 but on the timestamp path.

### What v2.3.4 does

Pre-parse string guard before `new Date()`:

```js
function _luRaw_(raw) {
  if (raw == null) return null;
  const s = String(raw).trim();
  if (!s) return null;
  const lo = s.toLowerCase();
  if (lo === 'null' || lo === 'undefined' || lo === 'invalid date') return null;
  return s;
}
function _luMs_(raw) {
  const s = _luRaw_(raw);
  if (!s) return null;
  const t = new Date(s).getTime();
  return isFinite(t) ? t : null;
}
```

Then in `refreshLive()`:

```js
const ageMs = _luMs_(generatedAt);
const ageMin = ageMs == null ? null : (Date.now() - ageMs) / 60000;
// Safe-fail: if we cannot prove freshness, HOLD staleKill (don't clear it).
if (ageMin == null || cbTripped || ageMin > LIVE_STALE_MIN) {
  // keep staleKill flag
} else {
  setStaleKill(false);
}
```

### Validation receipts

- `med5_iso_string_ok: ok` — `'2026-06-24T12:00:00Z'` parses
- `med5_null_literal_rejected: ok` — `'null'` → `null`
- `med5_NULL_uppercase_rejected: ok` — case-insensitive
- `med5_undefined_literal_rejected: ok`

## MED #6 — `refreshOutrights()` clobbered manually-entered formulas in col L

### What was wrong (v2.3.3)

`refreshOutrights()` read col L (operator override notes / IMPORTRANGE links) via `getValues()`, which silently strips the leading `=` and returns the computed value. Re-writing the snapshot then committed values back over the original `=IMPORTRANGE(...)` formulas, permanently destroying the formula on every refresh tick.

### What v2.3.4 does

Parallel snapshot of both formulas and values; formula wins when present:

```js
const valSnap = rng.getValues();
const formSnap = rng.getFormulas();
for (let i = 0; i < n; i++) {
  if (formSnap[i] && formSnap[i][0]) {
    restore[i] = { mode: 'formula', payload: formSnap[i][0] };
  } else if (valSnap[i] && valSnap[i][0] !== '' && valSnap[i][0] != null) {
    restore[i] = { mode: 'value', payload: valSnap[i][0] };
  } else {
    restore[i] = { mode: 'none', payload: null };
  }
}
```

On write-back, `setFormula()` is used for `mode==='formula'`, `setValue()` for `mode==='value'`, and `none` rows are left blank (new teams).

### Validation receipts

- `med6_formula_wins_over_value: ok` — given `formulas[i] === '=IMPORTRANGE(...)'` and `values[i] === 42`, restore payload is the formula
- `med6_value_used_when_no_formula: ok` — given empty formula and `values[i] === 6`, restore mode is `'value'` payload `6`
- `med6_no_restore_for_new_team: ok` — both empty, mode is `'none'`
