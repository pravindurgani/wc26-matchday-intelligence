# WC26 Engine v2.3.2 — Patch Notes

**Release type:** single-CRIT bug-fix on top of v2.3.1.
**Schema/XLSX changes:** none. Drop-in replacement.
**Compatible with:** `https://wc26-matchday-intelligence.vercel.app/` (Rounds 11–16.1).

## CRIT #1 — `refreshIntel()` silently dropped 48/61 active adjustments

### What was wrong (v2.3.1)

The pre-fix loop in `refreshIntel()` did:

```js
const mid = Number(a.match_id);
if (!isFinite(mid)) return;     // <-- silent drop of every team-level entry
```

The Python canonical that writes `/matchday_intelligence.json` buckets
**injury** and **stats_proxy** components under `(team, match_id=null)` —
see `scripts/live/apply_matchday_adjustments.py:1040-1048`. Those entries
are tournament-wide: the official adjuster
`get_team_elo_adjustment(team, match_id)` at
`apply_matchday_adjustments.py:1480-1504` applies them to **every** match
where that team plays, by treating `entry.match_id is None` as a
wildcard:

```python
if entry["match_id"] is None or entry["match_id"] == match_id:
    total += entry["total_elo_adjustment"]
```

So on the live snapshot (61 active components — 48 of them team-level
injury / stats_proxy, 13 of them match-level weather), Apps Script was
emitting an intel-delta of **0** for the 72 group-stage matches whose
only context signal was a team-level adjustment (e.g. Belgium −22.6
elo from De Bruyne+Lukaku doubts, Algeria −11.08 from form proxy).
The Bets sheet stake decisioning therefore disagreed with the live
dashboard for those 72 matches.

### What v2.3.2 does

`refreshIntel()` (lines ~460–535 of `WC26_Engine_AppsScript_v2.3.2.gs`)
now:

1. Pre-fetches `predictions_live.match_predictions` once and builds a
   `matchTeams[m] → {home, away}` index.
2. Iterates `intel.active_adjustments`. For each entry:
   - If `a.match_id` is finite → bucket each component's `capped_elo`
     under that single match (matches v2.3.1 behaviour for
     weather rows). Falls back to `total_elo_adjustment` if the
     `components[]` array is empty (cap-suppressed edge case).
   - If `a.match_id` is null → loop over `matchTeams` and credit
     `total_elo_adjustment` to every match where `team` plays home or
     away. The breakdown column gets a single `'<team> team-wide: ±N'`
     line to keep the cell readable.
3. Wraps the predictions fetch in `try/catch`. On fetch failure, the
   loop falls back to match-level-only (== v2.3.1 behaviour) and
   surfaces the cause in the Live!intelWarn cell so the operator sees
   the degradation.

### Validation

Reran a Python mirror of the new logic over the live snapshot:

- **72 matches** now carry non-zero intel deltas (up from ≤ 13 pre-fix).
- **0 mismatches** versus the canonical
  `apply_matchday_adjustments.get_team_elo_adjustment(team, m_id)` —
  sum of |delta| across the 72 matches matches to the rounding digit.
- Engine math unaffected: `test_harness.mjs` (`node v24`,
  `GOAL_GRID_MAX_GOALS=15`, `GOAL_GRID_TAU=-0.13`) still returns:
  - All three λ scenarios sum to **Σ = 1.0**.
  - Symmetric (1.4, 1.4): `home = away = 0.3581`, `draw = 0.2838`
    (DC-τ uplift on the draw mass intact).
  - Asymmetric swap mirror exact: `home(1.8,0.9) = away(0.9,1.8) = 0.5617`.
  - Knockout classifier: `R32:16 + R16:8 + QF:4 + SF:2 + 3rd:1 + Final:1
    = 32`. `m=72` and `m=105` both return `null`.
