**Release type:** seven-closure hardening sweep on top of v2.3.4 — two P0 (would-not-stake on the day), two HIGH, two MED, one P1.
**Schema/XLSX changes:** none. Drop-in replacement.
**Compatible with:** `https://wc26-matchday-intelligence.vercel.app/` (Rounds 11–16.1, hardening R14).

## P0-A — `extendToKnockouts()` only seeded `match_no` + BC:BE — staking matrix was blank for rows 74–105

### What was wrong (v2.3.4 and earlier)

The Bets sheet ships with **xlsx shared formulas** scoped to the group-stage window. Every staking cell on row 2 carries a `ref="R2:R73"` attribute (verified by `xl/worksheets/sheet4.xml`). xlsx shared formulas do NOT autofill into rows 74–105. Pre-v2.3.5 `extendToKnockouts()` only wrote two things:

1. The `match_no` column (A) — values 73…104.
2. Blank-safe mirror formulas in BC:BE (the calibrated probability block — v2.3.1 CRIT #3 / v2.3.3 HIGH #5).

The full per-row staking matrix — fair probabilities (O,P,Q), book sum + devig (R,S,T,U), blend (V,W,X), EV (Y,Z,AA), pick (AB,AC,AD,AE,AF,AG), decision (AH), Kelly stake (AI,AJ), chosen blend (AK,AL,AM), Matchday-mirror (AN,AO,AP,AQ), settlement (AR,AS,AT,AU,AV), and snap-or-current (AZ,BA) — was **left completely blank on rows 74–105**.

Net effect: had v2.3.4 shipped to R32 kickoff at 2026-06-28, every knockout match would have rendered as a row with model probabilities (B:K populated by `refreshModel`) but **no devig, no EV, no pick, no engine stake**. The Bets sheet would silently fail to stake any of the 32 knockout matches.

### What v2.3.5 does

1. Adds `_knockoutStakingFormulas_(r)` (lines 1520–1571) — a pure-logic helper that returns three contiguous formula blocks for a given row `r`:
   - `O_AQ` (29 cells, cols O..AQ): fair probs → book sum → devig → blend → EV → max-EV idx → pick label/prob/EV/edge → decision → Kelly % → Kelly stake → chosen blend → Matchday-mirror (AN/AO/AP/AQ).
   - `AR_AV` (5 cells, cols AR..AV): settlement P&L → running P&L → message → audit row → backed-pick mirror.
   - `AZ_BA` (2 cells, cols AZ..BA): snap-or-current decision and stake.
   - All references are per-row (`L{r}`, `Method!$B$N`, `Matchday!N{rMatchday}` with `rMatchday = r + 2` — Matchday has two extra header rows above Bets).
2. Wires the helper into `extendToKnockouts()` (lines 1647–1707) with **the same formula-preservation pattern as the existing BC:BE seed**: read the block via `getFormulas()`, skip the row if any cell already carries a formula (operator hand-edit), else seed the whole block. This is idempotent — running the menu item twice never overwrites existing work.
3. Status message now reports per-block seeded counts: `staking O..AQ=32 AR..AV=32 AZ..BA=32 row(s) seeded`.

The three blocks intentionally skip cols **AW/AX/AY** (operator-typed snap cells for placed-bet capture). Those remain blank by design.

## P0-B — `_writeCalibratedProbs_` applied the group-stage isotonic LUT to knockout rows

### What was wrong (v2.3.4)

The calibration LUT (`/data/processed/wc26_calibration.json` on the Vercel feed) was trained on **group-stage results only**. The draw market in particular has no representation for knockout matches — they don't draw within 90 minutes by their nature (extra time / penalties replace the group-stage draw mass). Pre-v2.3.5, `_writeCalibratedProbs_` applied the same 4-bin LUT to all 104 rows including knockouts. The corrected draw probability written into BD74..BD105 was a **statistical fiction** that downstream EV math then traded against.

### What v2.3.5 does

`_writeCalibratedProbs_` now accepts an explicit `matchNos` parameter (threaded from `refreshModel` at line 466). For every row in the window, it checks `m >= KNOCKOUT_FIRST_M (73)`:

```js
if (isFinite(m) && m >= KNOCKOUT_FIRST_M) {
  koSkipped++;
  return [_safe_(row[0]), _safe_(row[1]), _safe_(row[2])];  // raw mirror, no LUT
}
```

Knockout rows are raw-mirrored (BC..BE = I..K) instead of LUT-corrected. The skip count is surfaced in the status message and in the dedicated `Live!calibrationOob` cell (B27 — see H-2 below).

## HIGH H-1 — fallback path could pass NaN into `setValues`; Apps Script throws

### What was wrong (v2.3.4)

The fallback branch inside `_writeCalibratedProbs_` (v2.3.4 HIGH #3) used `Number(row[0])` to coerce the raw probability when `_interp_` returned `''`. If the source cell was actually `undefined` (newly inserted row, never written), `Number(undefined) === NaN`. Apps Script `Range.setValues()` throws `Cannot use value: NaN` on any NaN cell, killing the whole `refreshModel` tick. The error message lands in `Logger.log` but the operator sees a stale Live state.

### What v2.3.5 does

Introduces a `_safe_(v)` helper inline:

```js
function _safe_(v) {
  if (v === '' || v === null || v === undefined) return '';
  const n = Number(v);
  if (!isFinite(n) || isNaN(n)) return '';
  return n;
}
```

Every return path in `_writeCalibratedProbs_` now routes through `_safe_(...)`. The fallback, the OOB clamp, and the knockout raw-mirror all guarantee the returned cell is either a finite number or `''` — never NaN, never undefined.

## HIGH H-2 — OOB count overwrote `Live!intelWarn` and raced `refreshIntel`

### What was wrong (v2.3.4)

v2.3.4 HIGH #4 surfaced the binned-LUT out-of-range clamp count via `Live!intelWarn` (B24). That cell is also written by `refreshIntel()` and by `extendToKnockouts()`'s conflict tagger. On any tick where both fired, whichever ran last overwrote the other — the operator either saw the OOB count or the intel-fetch warning, never both.

### What v2.3.5 does

1. Added `LIVE_CELL.calibrationOob = 'B27'` (engine line 313).
2. `_writeCalibratedProbs_` writes the OOB summary to `LIVE_CELL.calibrationOob`, NOT `intelWarn`.
3. `intelWarn` is now owned exclusively by `refreshIntel()` and the `extendToKnockouts()` conflict tagger — no contention.

## MED M-1 — OOB count had no denominator; "home=2" was ambiguous

### What was wrong (v2.3.4)

The OOB warning rendered as `home=2 draw=5 away=1` — no row-count context. The operator couldn't tell whether 2 OOB clamps out of 48 group-stage rows is normal noise or 2 out of 3 (a calibration failure).

### What v2.3.5 does

Now renders `home=2/48 draw=5/48 away=1/48` — every count carries the denominator `denom = window-row count`. The source idiom is `'home=' + oobCounts.home + '/' + denom + ' draw=' + ...`.

## MED M-2 — missing-timestamp branch suppressed circuit-breaker signal

### What was wrong (v2.3.4)

`refreshLive()` had two independent stale-data conditions: (a) `last_updated_utc` missing/invalid, (b) upstream circuit-breaker tripped. When both fired simultaneously, the missing-timestamp branch wrote a generic "model paused" message and returned, never surfacing that the CB was also tripped. The operator would assume the dashboard publisher silently dropped the timestamp field and waste investigation time on the wrong side.

### What v2.3.5 does

The missing-timestamp branch now checks `cbTripped` too:

```js
if (cbTripped) {
  staleMsg = '⚠ circuit-breaker tripped upstream AND feed missing last_updated_utc — model paused';
} else {
  staleMsg = '⚠ feed missing last_updated_utc — model paused (safe-fail)';
}
```

## P1 — outrights sort was non-deterministic on ties

### What was wrong (v2.3.4)

`refreshOutrights()` sorted teams by `p_champion` descending with no tiebreak. Two teams tied at the rounding digit (a common state: WC26 has 48 entrants and the LUT publishes 3 decimal places, so ~3-5 pairs collide every tick) sorted in V8's `Array.sort` order — implementation-defined and non-stable. The same data could render in different orders across successive refreshes, confusing the operator.

### What v2.3.5 does

Sort comparator now has a `localeCompare(team)` secondary key:

```js
teams.sort(function(a, b) {
  const d = _num_(b.p_champion) - _num_(a.p_champion);
  if (d !== 0) return d;
  return String(a.team || '').localeCompare(String(b.team || ''));
});
```

## Operator runbook (R32 install)

1. Pull `WC26_Engine_AppsScript_v2.3.5.gs` into the Apps Script project (replace the existing file in place — same script ID).
2. From the Sheet menu, **WC26 Engine → Extend to knockout stage (run once before R32)**. Verify the status toast reads `staking O..AQ=32 AR..AV=32 AZ..BA=32 row(s) seeded`. The 32 is the count of newly-seeded rows; a re-run will show 0 because the preservation guard skips already-populated cells.
3. Spot-check Bets row 74 (m=73): `O74` should be a fair probability `1/L74` value (or `''` if L74 is empty). `AN74` should mirror `Matchday!N76`.
4. From the Sheet menu, **WC26 Engine → Refresh everything**. Confirm Bets row 74 onwards renders devig / EV / pick / decision / stake columns.
5. Confirm `Live!B27` is blank or carries a sensible OOB count like `home=0/72 draw=0/72 away=0/72 (ko_skipped=32 of 32)`.

If step 2 surfaces a `CONFLICT` count > 0, an operator hand-edit on rows 74–105 holds a non-knockout `match_no` value. Inspect those rows manually before relying on knockout staking.
